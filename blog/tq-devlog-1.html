Devlog |#1: postgres, docker, beekeeper, lazygit​

Another day I actually started working on the project (what a surprise).

I want to setup my repository, basic development environment, something to make it cool and get a chance to procrastinate a bit more.

Im convinced I will store my data in postgres, therefore I need a local instance, migrations, a tool to lookup the data.

First, docker compose. I can't imagine local environment without it. Very simple one, some volumes, passwordless acceess, expose 5432 port, done.

I didn't know how my first feature will look like, a user creating, connecting a repository or saving the infrastracture state. But I needed a first migration to test it, so I created a users table. Very creative, but I really will need users table, it's fine.

I installed a few dependencies:

go get github.com/golang-migrate/migrate/v4
go get github.com/jmoiron/sqlx
go get github.com/jackc/pgx

I will sqlx + squirrel for query executing, pgx as a driver and migrate for running migrations, it prodes cli interface if you want to run in the jobs and go api as well.

I will use go api, but I wouldn't recomment it in production, it's important to create a database backup before starting a mgiration, so the best case you have a migration job that makes a backup and then runs the migration, if it worked out - release is welcome.

Then I made a few commands to work with migration like create new, up, down and fix migration.

note

Migration package stores the state in the same database. When it fails it writes the migration as dirty. Fix command is used to force the version, it's very useful in the development process. I even used it in prod.

DB_DSN ?= "postgres://postgres@localhost:5432/tq?sslmode=disable"

migrate_new:
    migrate create -ext sql -dir migrations -seq -digits 4 ${MNAME}

migrate_up:
    migrate -path migrations -database ${DB_DSN} up

migrate_down:
    migrate -path migrations -database ${DB_DSN} down

migrate_fix:
    migrate -path migrations -database ${DB_DSN} force ${V}

migrate_v:
    migrate -path migrations -database ${DB_DSN} version

Using command make migrate_new users_table I create a new migration and feel it with the following sql:

CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

CREATE TABLE IF NOT EXISTS users (
    id uuid PRIMARY KEY DEFAULT uuid_generate_v4() NOT NULL,
    email varchar(85) NOT NULL,
    password varchar(255) NOT NULL,

    createdAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP NOT NULL,
    updatedAt TIMESTAMP DEFAULT CURRENT_TIMESTAMP NOT NULL
);

I like using uuid as an identity. It never fooled me.

Then I do what I didn't recommend.

Yes, I run the migrations right in the starting the app server. Seems ok, applied.

I need something to see the result, the table, the extension. I use psql, but to me personnaly it's very annoying having such long interface, it matters a lot when I have a few databases to follow.

I looked for a tool again. I tried pgadmin, jetbrains embedded one. And I found Beekeeper. That's the tool I wouldn't be ashamed to create. Looks very good, easy to use, to edit, look for data with 0 sql, comfortable keyboard control. Everything I like. It's even opensourced and has a free community edition.

I looked at the data, the migration record exists, the users table does as well.

Finally I can commit it. I like using lazygit for most of my cases. It also a keyboard only control. Even my brain can handle it, there are very few commands: arrows to navigate, space to stage a change, you can use mouse to stage only a part of the file, and push c to input a commit message and Enter to commit. It also has bunch of things like branches list, reflog, command log.

At least I don't repeatadly do anymore ls, gd, ga file by file.

note

I use zsh and it gives me many shortcuts, for instance ga is equal to git add

Thank you for reading, come here and see what's gonna be in the end. Repository is here: http://github.com/treenq/treenq
